## Quick Start

### Minimal Example

```swift
import Colony

// Create a runtime with default settings
let runtime = try ColonyBuilder()
    .model(name: "llama3.2")
    .profile(.device)
    .capabilities([.planning, .filesystem])
    .build()

// Start a conversation
let handle = await runtime.runControl.start(
    .init(input: "Read the files in /project and summarize them")
)

// Wait for completion
let outcome = try await handle.outcome.value

// Extract the answer
if case let .finished(output, _) = outcome {
    let answer = try await output.store.get(ColonySchema.Channels.finalAnswer)
    print(answer)
}
```

### With Tool Approval (Human-in-the-Loop)

```swift
import Colony

let runtime = try ColonyBuilder()
    .model(name: "claude-sonnet")
    .profile(.cloud)
    .capabilities([.planning, .filesystem, .shell])
    .toolApprovalPolicy(.always)  // All tools require approval
    .build()

let handle = await runtime.runControl.start(
    .init(input: "Create a new Swift file that says hello")
)

// When interrupted for approval...
if case let .interrupted(interrupt) = outcome {
    // Show user the tool calls
    if case let .toolApprovalRequired(toolCalls) = interrupt.interrupt.payload {
        for toolCall in toolCalls {
            print("\(toolCall.name): \(toolCall.arguments)")
        }

        // User approves or rejects
        let decision: ColonyToolApprovalDecision = .approved  // or .rejected
        runtime.resumeToolApproval(
            interruptID: interrupt.interrupt.id,
            decision: decision
        )
    }
}
```

## Architecture Overview

Colony is an agent framework that orchestrates a cyclic runtime loop where an LLM repeatedly calls tools until it produces a final answer or an interrupt fires. The framework is built as a thin orchestration layer on top of **HiveCore**, which provides the core runtime, graph compilation, and checkpointing primitives.

### Key Distinguishing Characteristics

1. **Capability-gated tool families** - Tools are only injected when both the capability is enabled AND a backend is wired
2. **Two-tier approval system** - Policy-level (`.never`/`.always`/`.allowList`) + fine-grained rule-level with pattern matching
3. **Cryptographic audit chain** - Tool approvals logged with HMAC-SHA256 signatures in a hash chain for tamper evidence
4. **Token budget-aware prompt assembly** - Scratchbook placed BEFORE memory/skills in prompt for truncation resilience
5. **Virtual filesystem** - All operations use normalized virtual paths, never raw filesystem paths
6. **Composite filesystem** - Multiple backends mounted by path prefix (like Unix filesystem)

## The Cyclic State Machine - Runtime Loop

The heart of Colony is a cyclic state machine defined in `ColonyAgent.swift`:

```
preModel → model → routeAfterModel → tools → toolExecute → preModel
```

### Node Definitions

| Node | Purpose |
|------|---------|
| `nodePreModel` | Compacts messages if needed, applies summarization policy, prepares LLM input |
| `nodeModel` | Calls the LLM with tools and receives tool calls or final answer |
| `nodeRouteAfterModel` | No-op router that checks pending tool calls - routes to tools if present |
| `nodeTools` | Prepares tool execution, checks approval policy, emits interrupts if needed |
| `nodeToolExecute` | Executes approved tools, writes results as assistant messages, loops back |

### Channel Flow

1. **preModel** reads from `messages`, compacts/summarizes, writes to `llmInputMessages`
2. **model** reads `llmInputMessages` + tools, writes tool calls to `pendingToolCalls` OR final answer to `finalAnswer`
3. **routeAfterModel** checks `pendingToolCalls` - if non-empty, routes to `tools`; if empty, loop terminates
4. **tools** assesses each tool call for approval requirement via `ColonyToolSafetyPolicyEngine`
5. **toolExecute** runs approved tools, writes results as `assistant` role messages with `toolCallId`, returns to `preModel`

## ColonySchema - Data Channels

The schema defines all agent data channels:

| Channel | Type | Scope | Purpose |
|---------|------|-------|---------|
| `messages` | `[HiveChatMessage]` | global | All conversation messages |
| `llmInputMessages` | `[HiveChatMessage]?` | global | Messages being sent to LLM (compacted view) |
| `pendingToolCalls` | `[HiveToolCall]` | global | Tool calls awaiting execution |
| `finalAnswer` | `String?` | global | Final response from agent |
| `todos` | `[ColonyTodo]` | global | Task tracking todos |
| `currentToolCall` | `HiveToolCall` | task-local | Currently executing tool call |

### Risk Levels (Ordered by Severity)

```swift
enum ColonyToolRiskLevel: Int, Comparable {
    case readOnly = 0        // No side effects (ls, read_file, glob, grep)
    case stateMutation = 1  // Modifies internal state (scratchpad, todos)
    case mutation = 2        // Modifies files, disk, git (write_file, edit_file)
    case execution = 3       // Arbitrary code/command execution (execute, task)
    case network = 4         // External network requests (git_push, web_search)
}
```

## Tool Approval and Safety

### Two-Tier Approval System

**Tier 1: Policy-Level** (`ColonyToolApprovalPolicy`)
- `.never` - all tools auto-execute
- `.always` - all require approval
- `.allowList(Set<String>)` - only non-listed require approval

**Tier 2: Rule-Level** (`ColonyToolApprovalRuleStore`)
- Pattern matching: `.exact(String)`, `.prefix(String)`, `.glob(String)` (with `*` and `?`)
- Decision types: `.allowOnce` (consumed after use), `.allowAlways`, `.rejectAlways`
- Priority: exact > prefix > glob, then by recency

### ColonyToolSafetyPolicyEngine

Evaluates tool safety and determines approval requirements:

```swift
struct ColonyToolSafetyAssessment {
    var toolCallID: String
    var toolName: String
    var riskLevel: ColonyToolRiskLevel
    var requiresApproval: Bool
    var reason: ColonyToolApprovalRequirementReason
}

enum ColonyToolApprovalRequirementReason {
    case mandatoryRiskLevel           // Risk level mandates approval
    case policyAlways                 // Policy requires all tools
    case policyNotAllowListed         // Tool not in allow list
}
```

### Cryptographic Audit Chain

Tool approvals are logged with cryptographic signatures forming a hash chain:

```swift
struct ColonySignedToolAuditRecord {
    var entryID: String
    var previousEntryHash: String     // SHA-256 of previous entry
    var payload: ColonyToolAuditPayload
    var entryHash: String              // SHA-256(chainData)
    var signature: String              // HMAC-SHA256(entryHash, key)
}

// Chain data format: "previousEntryHash\n" + payloadJSON
// Entry hash = SHA256(chainData)
// Signature = HMAC-SHA256(entryHash, key)
```

Verification checks: sequence continuity, hash chain integrity, signature authenticity.

## Memory and Scratchbook

### ColonyMemory

Simple term-overlap scoring for memory search:

```swift
protocol ColonyMemoryService {
    func search(_ request: ColonyMemorySearchRequest) async throws -> ColonyMemorySearchResponse
    func store(_ request: ColonyMemoryStoreRequest) async throws -> ColonyMemoryStoreResponse

// Scoring algorithm:
// - Case-insensitive token matching

// - Falls back to substring match (0.25 score) if no overlap
// - Results sorted by relevance score, then by ID
```

### ColonyWorkspace (Scratchbook)

Replaces the deprecated `ColonyScratchbook`. Stores notes, todos, and tasks:

```swift
struct ColonyWorkspaceItem {
    var id: String
    var kind: Kind          // .note, .todo, .task
    var status: Status      // .open, .inProgress, .blocked, .done, .archived
    var title: String
    var content: String?
    var tags: Set<String>
    var phase: String?
    var progress: Double?
    var createdAt: UInt64   // Nanoseconds since epoch
    var updatedAt: UInt64
    var pinned: Bool
}

// Status aliases: active = .open/.inProgress/.blocked, pending = .open, completed = .done/.archived
```

**View Rendering Priority:**
1. Pinned items first
2. Then tasks sorted by: status priority → updated timestamp
3. Then todos sorted by: status priority → updated timestamp
4. Then notes sorted by: updated timestamp

**Token Budget Handling:**
- Scratchbook placed BEFORE memory/skills in prompt (for truncation resilience)
- `viewTokenLimit` (default 800) limits scratchbook rendering
- `maxRenderedItems` (default 40) limits displayed items
- Approximation: `tokenLimit * 4` characters

### Scratchbook Filesystem Persistence

- Thread-scoped: each thread has its own scratchbook file
- Path: `/scratchbook/{threadID_sanitized}.json`
- Filename sanitization: converts to safe ASCII
- Atomic writes with edit fallback

## Interrupt/Resume Cycle

### When Interrupts Fire

The runtime emits `.interrupted` outcome when:
- Tool requires approval (via `ColonyToolSafetyPolicyEngine`)
- No resume payload present in the run request

### Interrupt Flow Diagram

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Runtime Loop                                   │
│                                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐    ┌───────────┐ │
│  │ preModel │───▶│  model  │───▶│routeAfterModel│───▶│   tools  │ │
│  └──────────┘    └──────────┘    └──────────────┘    └───────────┘ │
│                                                    │                │
│                                                    ▼                │
│                                            ┌─────────────┐          │
│                                            │   needs     │          │
│                                            │ approval?   │          │
│                                            └─────────────┘          │
│                                               │      │               │
│                                              YES     NO              │
│                                               │      │               │
│                                               ▼      └───────┐       │
│                                    ┌─────────────────┐      │       │
│                                    │   .interrupted  │      │       │

│                                    └────────┬────────┘      │       │
│                                             │               │       │
└─────────────────────────────────────────────┼───────────────┼───────┘
                                              │               │
                                              ▼               │
                               ┌──────────────────────────┐  │
                               │   External Handler       │  │
                               │   (Human in the Loop)   │  │
                               │                          │  │
                               │  Show tool to user:      │  │
                               │  - Name: write_file      │  │
                               │  - Args: {"path":...}   │  │
                               │                          │  │

                               │  [y] Approve             │  │

              │    .approved)       │               │    .rejected)      │
              └──────────┬──────────┘               └──────────┬──────────┘

                                   └──────────────┘
```

### Interrupt Flow (Text)

1. Model returns tool call
2. tools node assesses via ColonyToolSafetyPolicyEngine
3. Approval required → emit .interrupted with .toolApprovalRequired payload
4. External handler prompts user (y/n/c per tool)
5. User approves/rejects via runtime.resume(..., payload: .toolApproval(decision:))
6. .approved: tool executes, result returned to model
7. .rejected: model receives "Tool execution rejected by user" system message

### Resume Payload

```swift

// or
ColonyResumePayload.toolApproval(interruptID:, perToolDecisions:)
// for per-tool decisions
```

```swift
@Test("Agent writes file after approval")
func testToolApprovalAndExecution() async throws {
    // 1. Setup - create mock model that returns a tool call
    final class WriteFileModel: HiveModelClient {
        func complete(_ request: HiveChatRequest) async throws -> HiveChatResponse {
            // Return a write_file tool call
            HiveChatResponse(
                content: [.text("I'll write the file.")],
                toolCalls: [
                    HiveToolCall(id: "call-1", name: "write_file",
                                 arguments: #"{"path":"/note.md","content":"hello"}"#)
                ]
            )
        }
    }

    // 2. Create runtime with filesystem
    let fs = ColonyInMemoryFileSystemBackend()
    let runtime = try ColonyBuilder()
        .model(AnyHiveModelClient(WriteFileModel()))
        .profile(.device)
        .capabilities([.filesystem])
        .toolApprovalPolicy(.always)
        .filesystem(fs)
        .build()

    // 3. Start run - will interrupt for approval
    let handle = await runtime.runControl.start(.init(input: "Create a file"))

    // 4. Wait for interrupt
    let outcome = try await handle.outcome.value
    guard case let .interrupted(interrupt) = outcome,
          case let .toolApprovalRequired(toolCalls) = interrupt.interrupt.payload
    else {
        #expect(Bool(false), "Expected tool approval interrupt")
        return
    }

    // 5. Approve the tool
    let resumed = await runtime.runControl.resume(
        .init(interruptID: interrupt.interrupt.id, decision: .approved)
    )

    // 6. Verify outcome
    let finalOutcome = try await resumed.outcome.value
    #expect(finalOutcome.isFinished)

    // 7. Verify file was written
    let content = try await fs.read(at: ColonyVirtualPath("/note.md"))
    #expect(content == "hello")
}
```

## Key Design Patterns

### 1. Protocol-Based Backends

All major subsystems are protocol-based for testability and flexibility:
- `ColonyFileSystemService`, `ColonyShellService`, `ColonySubagentRegistry`, `ColonyMemoryService`, `ColonyLSPService`, `ColonyGitService`

### 2. Actor Isolation

In-memory backends use actors for thread safety:
- `ColonyInMemoryMemoryBackend`, `ColonyInMemoryFileSystemBackend`, `RecordingShellBackend`, `RecordingSubagentRegistry`

### 3. Immutable Builder Pattern

`ColonyBuilder` returns new copies on each method call:

```swift
let builder1 = ColonyBuilder()
let builder2 = builder1.model(name: "test-model")
// builder1 is unchanged
```

### 4. Deprecated Type Aliases

Heavy use of `@available(*, deprecated, renamed:)` for backward compatibility:
- `ColonyScratchbook` → `ColonyWorkspace`
- `ColonyAgentFactory` → `ColonyBuilder`
- `onDevice4k` → `device`

### 5. Virtual Paths

All filesystem operations use virtual paths, never raw filesystem paths:
- No `..`, no `~`, normalized absolute paths
- Standard roots: `/`, `/scratchbook`, `/conversation_history`, `/audit/tool_decisions`

### 6. Composite Pattern

`ColonyCompositeFileSystemService` composes multiple backends:
- Longest-prefix matching for route resolution
- Virtual route directories automatically exposed

### 7. Token Budget Awareness

Scratchbook placed BEFORE memory/skills in prompts for truncation resilience:
- When context is hard-truncated, active state survives
- View ordering optimized for most critical information first

### 8. Cryptographic Audit Chain

Tool approvals logged with HMAC-SHA256 signatures forming a hash chain:
- Tamper-evident audit trail
- Blockchain-style entry linking

### 9. Typed ID System

Phantom-domain identifiers for type safety:
- `ColonyThreadID`, `ColonyRunID`, `ColonyRunAttemptID`, `ColonyCheckpointID`, `ColonyInterruptID`, `ColonyChannelID`, `ColonyNodeID`, `ColonySubagentType`, `ColonyArtifactID`

### 10. Capability Gating

Tools injected only when BOTH:
1. Capability enabled in configuration
2. Backend wired in environment

## Troubleshooting

### Common Issues

#### 1. Runtime Won't Start - Model Not Found

**Symptom:** `HiveError.modelNotFound` or similar

**Solution:** Ensure the model name matches exactly what's registered:
```swift
// Check available models
let runtime = try ColonyBuilder()
    .model(name: "claude-3-5-sonnet")  // Must match exactly
    .build()
```

#### 2. Tools Not Available

**Symptom:** Model can't find expected tools (e.g., `ls`, `read_file`)

**Diagnosis:** Check if capability is enabled AND backend is wired:

```swift
// Capability must be enabled
.capabilities([.filesystem])  // .filesystem must be present

// And backend must be wired (automatic for defaults, but for custom:)
.filesystem(myCustomFS)  // If using custom backend
```

#### 3. Tool Approval Never Triggers

**Symptom:** Expected interrupt doesn't fire

**Solution:** Check the tool approval policy and risk levels:

```swift
// For tools to require approval:
// 1. Policy must not be .never
.toolApprovalPolicy(.always)  // or .allowList

// 2. Tool's risk level must be in mandatoryApprovalRisks (default: .mutation, .execution, .network)
.toolApprovalPolicy(.always)  // All tools require approval
.mandatoryApprovalRisks([.execution])  // Even read-only tools
```

#### 4. Out of Memory / Token Limits

**Symptom:** `ColonyBudgetError` or truncated responses

**Solution:** Use a profile with appropriate limits:

```swift
// For constrained environments
.profile(.device)  // 4k token budget

// Or customize explicitly
ColonyConfiguration(
    requestHardTokenLimit: 4000,
    compactionThreshold: 2600
)
```

#### 5. Subagent Recursion Error

**Symptom:** `ColonySubagentError.recursiveSubagentNotAllowed`

**Cause:** A subagent tried to spawn another subagent (by default disabled)

**Solution:** This is intentional - nested subagents must be explicitly enabled:

```swift
// Default subagent config disables recursive .subagents
// To allow nested subagents (use with caution):
var subagentConfig = ColonySubagentConfiguration()
subagentConfig.capabilities.insert(.subagents)
```

## FAQ

### Q: How does Colony differ from other agent frameworks like LangChain or AutoGPT?

**A:** Colony is designed specifically for Apple's platforms (iOS 26+/macOS 26+) with:
- Native Foundation Models integration (on-device inference)
- HiveCore dependency for deterministic runtime
- Capability-based tool gating (tools only available when both capability AND backend exist)
- Cryptographic audit chain for compliance
- Two-tier approval (policy + fine-grained rules)

### Q: Can Colony run without network access?

**A:** Yes, with the `.device` profile:
```swift
.profile(.device)  // Uses on-device Foundation Models
```

### Q: How does checkpointing work?

**A:** Colony uses HiveCore's checkpoint system:
- Checkpoints save complete runtime state (messages, todos, tool calls)
- On interrupt, state is persisted
- On resume, state is restored from checkpoint

```swift
// Checkpoint on every interrupt (default)
let handle = await runtime.run(threadID:, input:, options:
    HiveRunOptions(checkpointPolicy: .onInterrupt)
)
```

### Q: What's the difference between `.cloud` and `.device` profiles?

**A:**

| Setting | `.device` | `.cloud` |
|---------|-----------|----------|
| Token budget | ~4,000 | ~200,000 |
| Compaction | 2,600 tokens | 12,000 tokens |
| Summarization | 3,200 tokens | 170,000 tokens |
| Tool eviction | 700 tokens | 20,000 tokens |
| Scratchbook | Enabled | Disabled |
| Approval | AllowList | Never |

### Q: Can I add custom tools?

**A:** Yes, implement `ColonyToolDefinition` and register:

```swift
let myTool = ColonyToolDefinition(
    name: "my_custom_tool",
    description: "Does something useful",
    parameters: [...],
    execute: { params in
        // Your implementation
    }
)
```

### Q: How does memory work with multiple threads?

**A:** Each thread has isolated state:
- `messages` are thread-scoped
- `scratchbook` persists per-thread via filesystem
- Memory service is shared (if using same backend)

### Q: What's the maximum conversation length?

**A:** Depends on profile and compaction settings:
- `.device`: ~4,000 tokens total (compacts aggressively)
- `.cloud`: ~200,000 tokens (with summarization at 170,000)
